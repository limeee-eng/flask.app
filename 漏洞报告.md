# 安全漏洞审计报告

## 项目：用户信息管理平台

> 严重等级：🔴 严重 | 🟠 高危 | 🟡 中危 | ⚪ 低危

---

## 🔴 严重漏洞

### V1 — 硬编码密钥导致 Session 伪造（CWE-798）

**文件：** `app.py:4`

```python
app.secret_key = "dev-key-2025"
```

**描述：** Flask 的 `secret_key` 用于签名 session cookie。使用硬编码的弱密钥（"dev-key-2025"），攻击者可以直接通过 `flask-unsign` 工具解码并伪造任意 session cookie，无需密码即可登录管理员账户。

**复现步骤：**
```bash
pip install flask-unsign
flask-unsign --decode --cookie '<session-cookie>'
flask-unsign --sign --cookie "{'username': 'admin'}" --secret 'dev-key-2025'
```

---

### V2 — 密码以明文存储（CWE-256 / CWE-312）

**文件：** `app.py:7-24`

```python
USERS = {
    "admin": {"password": "admin123", ...},
    "alice": {"password": "alice2025", ...},
}
```

**描述：** 密码以明文形式硬编码在源代码中。任何能读取源代码的人都能获取所有用户的密码凭据。

---

### V3 — 密码明文暴露到前端页面（CWE-200）

**文件：** `app.py:45` → `templates/index.html:10`

**描述：** 登录成功后，服务端将**包含密码的完整用户字典**传给模板，前端页面将密码渲染在 DOM 中。只要查看页面源代码或按 F12 即可看到登录用户的明文密码。

---

### V4 — Debug 模式开启交互式 Shell（CWE-489）

**文件：** `app.py:59`

```python
app.run(debug=True, host="0.0.0.0", port=5000)
```

**描述：** `debug=True` 启用 Werkzeug 调试器，会生成交互式控制台页面。访问 `/console` 路径可输入 PIN 码获得服务器端 Python 交互式 Shell，完全控制服务器。

---

### V5 — HTML 注释泄露管理员凭据（CWE-546）

**文件：** `templates/login.html:1`

```html
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```

**描述：** 在页面源代码中通过 HTML 注释直接公开了管理员用户名和密码。任何访问登录页面的用户都可以通过查看网页源代码立即获取管理员凭据。

---

## 🟠 高危漏洞

### V6 — 服务暴露到所有网络接口

**文件：** `app.py:59`

```python
app.run(debug=True, host="0.0.0.0", port=5000)
```

**描述：** `host="0.0.0.0"` 使服务监听所有网络接口（公网 IP、内网 IP、localhost）。如果部署在云服务器上，任何人可从公网访问管理平台。

---

### V7 — 登录接口无暴力破解防护

**文件：** `app.py:36-49`

**描述：** `/login` 没有任何速率限制、失败计数或账户锁定机制。攻击者可用脚本对任意用户名进行高速密码爆破。

---

### V8 — 无 CSRF Token 保护

**文件：** `templates/login.html:7`

**描述：** 登录表单没有 CSRF token，存在跨站请求伪造攻击风险。

---

### V9 — 敏感信息通过 Cookie 传输但无 HTTPS

**描述：** 所有 HTTP 流量（包括密码）均为明文传输。中间人攻击者可以嗅探网络流量，直接获取登录凭据和 session cookie。

---

## 🟡 中危漏洞

### V10 — 密码长度/复杂度无校验

**文件：** `app.py:39-40`

**描述：** 后端没有对密码做任何长度或复杂度校验。

---

### V11 — Jinja2 模板中直接渲染 session 变量

**文件：** `templates/base.html:16`

```html
<span class="nav-welcome">欢迎，{{ session['username'] }}</span>
```

**描述：** 直接在模板中渲染 `session['username']`。虽然 Jinja2 默认开启自动转义降低了 XSS 风险，但存在存储型 XSS 的架构隐患。

---

### V12 — 无 Session 过期机制

**文件：** `app.py`

**描述：** 没有设置 `PERMANENT_SESSION_LIFETIME`，session 永不过期，没有闲置超时和绝对超时。

---

## ⚪ 低危问题

### V13 — 响应时间侧信道

**文件：** `app.py:47`

**描述：** 登录成功和失败的响应时间可能因业务逻辑差异产生侧信道信息泄露，攻击者可据此辅助用户枚举。
